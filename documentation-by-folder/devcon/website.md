# website

Most of it is pretty straightforward / static page stuff.

Page layout is based on a global .section class that enforces width consistency between all pages - I'd familiarize myself with that one as it gets used basically everywhere to bound the layout (and also allow things to break free of the layout by adding .expand as a subclass). Rest is deliberately kept unabstracted to allow maximal flexibility (which is especially nice with AI exports from figma, just do whatever works, few rails).

Some things are legacy and esoteric - e.g. the header and footer are super messy - lean on AI is probably enough to figure it out if you ever want to refactor it.

# translations

Translations are generated by AI from the content folder - there's two types of translations, tina cms and i18n - I've been using i18n more at the cost of not having direct CMSability via tina - this is because tina operates in "markdown chunks" that make it extremely annoying/hard to implement more advanced sections - you can do it, but I recommend just dropping tina and go full i18n

Translations automatically run on any changes to the /content folder via a github action - it cleans up orphaned translations and keeps track of which files changed via content hashes so the AI pipeline only translates changes instead of the whole folder on every commit.

# faq translations

FAQs are pulled from nocodb via webhook, which triggers a github action, which writes to /content/external/faq - just as above, it gets re-translated automatically on change because its in the content folder - you'll have to cycle the github token as I am using my personal access token currently, it will probably stop working at some point

if you have other sources (e.g. make a cms in the future or whatever), you can put it in the content folder and it'll also get translated - its fully dynamic

# blog

the Devcon blog is pulled from the ethereum blog via RSS

you can fetch all our blogposts via the "Event" category, then you can further filter by the "Subcategory" which can be either Devcon or Devconnect - if this isn't working, its likely because the blog doesn't have the proper tags upstream in the blog repo - whoever handles the post submissions to the blog repo (historically Rose/Joe) would need to fix that / be aware of these categories - you can also create PRs there yourself, should be trivial to figure out how it works by inspecting the blog repo
